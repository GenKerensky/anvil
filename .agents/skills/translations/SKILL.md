# Translations

Guide for marking, generating, updating, and compiling translations in the Anvil extension. Based on [gjs.guide translations](https://gjs.guide/extensions/development/translations.html).

## How It Works

Translations are handled entirely by GNOME Shell's built-in gettext infrastructure:

1. `metadata.json` declares `"gettext-domain": "anvil"` — GNOME Shell auto-initializes translations when the extension is loaded. No manual `initTranslations()` call needed.
2. All user-visible strings in source files are wrapped in `_()` (the GNOME Shell `gettext` alias).
3. `xgettext` scans JS source files to extract those strings into a `.pot` template.
4. Translators create per-language `.po` files from the template.
5. `msgfmt` compiles `.po` → `.mo` binaries.
6. The `.mo` files are installed under `locale/<lang>/LC_MESSAGES/anvil.mo` inside the extension directory.

## Import Patterns

The `_()` function import path differs between extension and preferences code — they run in **separate processes**.

### Extension side (GNOME Shell process)

```js
import { Extension, gettext as _ } from "resource:///org/gnome/shell/extensions/extension.js";
```

Files using this import:

- `src/extension.ts`
- `src/lib/extension/indicator.ts`
- `src/lib/extension/window.ts`

### Preferences side (GTK4/Adwaita process)

```js
import { ExtensionPreferences, gettext as _ } from "resource:///org/gnome/Shell/Extensions/js/extensions/prefs.js";
```

Files using this import:

- `src/lib/prefs/widgets.ts:10`
- `src/lib/prefs/appearance.ts:7`
- `src/lib/prefs/floating.ts:5`
- `src/lib/prefs/settings.ts:14`
- `src/lib/prefs/keyboard.ts:7`

### Adding `_()` to a new file

Check whether the file runs in the extension process or preferences process to choose the correct import path. The `prefs.js` entry point itself does not import `_()` — only individual page files do, since they contain the user-visible strings.

## Marking Strings for Translation

### Simple strings

```js
_("Tiling");
_("Settings");
_("Border color");
```

### Strings with printf-style interpolation

Use `String.prototype.format()` (provided by GNOME Shell):

```js
_("Enabling %s").format(this.metadata.name);
```

### Strings with context (pgettext)

Use when a word has different meanings in different contexts (e.g. "Restart" as a window title vs. a button):

```js
import { gettext as _, pgettext } from "resource:///org/gnome/shell/extensions/extension.js";

pgettext("menu item", "Notify");
```

**Note**: `pgettext` is not currently used in Anvil, but the context argument is available if needed.

### Plural forms (ngettext)

For strings that vary by count in different languages:

```js
import { gettext as _, ngettext } from "resource:///org/gnome/shell/extensions/extension.js";

const body = ngettext("You have been notified %d time", "You have been notified %d times", count).format(count);
```

**Note**: `ngettext` is not currently used in Anvil, but is available if needed.

### What NOT to do

- Do NOT use `N_()` (no-op marker) — only the runtime `_()` function is used.
- Do NOT import `gettext` directly from `gi://GLib` — always use GNOME Shell's module exports.
- Do NOT use JavaScript template literals with `_()` — use `String.prototype.format()` with `%s`, `%d`, etc.

## Adding a New Translatable String

1. Wrap the string in `_()` in the source file.
2. Regenerate the POT template and merge translations:

```bash
make compilemsgs
```

This runs `xgettext` to scan for new strings → `src/po/anvil.pot`, then `msgmerge -U` to update each `.po` file with new/removed entries, then `msgfmt -c` to compile `.mo` files.

1. Rebuild the extension:

```bash
make build
```

This copies compiled `.mo` files into `dist/locale/<lang>/LC_MESSAGES/anvil.mo` as part of the standard extension layout.

## Build Pipeline (translation-specific)

From `Makefile:59-51`:

| Target        | Command                                                  | Effect                                                                      |
| ------------- | -------------------------------------------------------- | --------------------------------------------------------------------------- |
| `potfile`     | `xgettext --from-code=UTF-8 --output=src/po/anvil.pot`   | Scans `src/prefs.ts`, `src/extension.ts`, `src/lib/**/*.ts` for `_()` calls |
| `compilemsgs` | `msgmerge -U $lang.po src/po/anvil.pot` + `msgfmt -c`    | Updates each `.po` with new strings, compiles to `.mo`                      |
| `build`       | Copies `.mo` → `dist/locale/<lang>/LC_MESSAGES/anvil.mo` | Staged into extension output directory                                      |

`make build` depends on `compilemsgs`, so translations are always up-to-date in builds.

## Adding a New Language

1. Create a new `.po` file in `src/po/` using the template:

```bash
cp src/po/anvil.pot src/po/<lang_code>.po
```

1. Edit the `.po` file header (`Language`, `Plural-Forms`, etc.) and translate the `msgstr` entries using a tool like [Gtranslator](https://flathub.org/apps/details/org.gnome.Gtranslator) or [POEdit](https://flathub.org/apps/details/net.poedit.Poedit).

2. Run `make compilemsgs` to compile and merge.
3. Rebuild: `make build` (the new `.mo` will be included automatically since the Makefile matches `src/po/*.po` via `MSGSRC`).

## Testing Translations

### Unit tests

The `_()` function is mocked as a no-op identity function in unit tests:

- `test/unit/__mocks__/shell/extension.js:26-28` — `export function gettext(str) { return str; }`
- `vitest.config.js:24-27` — maps `resource:///org/gnome/shell/extensions/extension.js` to the mock

The prefs-side import path (`resource:///org/gnome/Shell/Extensions/js/extensions/prefs.js`) does **not** have a vitest mock. If testing prefs code that calls `_()`, add a mock for that path.

### E2E tests

E2E tests run in actual GNOME Shell (headless Wayland, via Podman). Translations are installed as part of `make dist`, which is used to build the E2E container image.

### Manual testing

```bash
# Force a locale for testing
LANGUAGE=es gnome-extensions prefs anvil@GenKerensky.github.com

# Or in looking glass:
# Alt+F2 → lg → Evaluator tab
_("Tiling")  # should return the translated string
```
