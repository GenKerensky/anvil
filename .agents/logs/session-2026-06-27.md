# Session Log — 2026-06-27

## Goal

Debug why some windows (specifically Inkscape main window) do not auto-tile on open. This is on the `codex/fix-delayed-window-tracking` branch.

## Symptom

- Launching Inkscape: its main document window opens but remains floating instead of being inserted into the active tiling layout.
- Other "some windows" occasionally exhibit the same.
- "Still" happening after prior delayed-tracking work.

## Investigation

- Primary entry: `display::window-created` → `WindowManager.trackWindow` (src/lib/extension/window.ts:325).
- `trackWindow` guards with `_validWindow` (accepts NORMAL/DIALOG/MODAL, rejects utils/popups + a few blacklisted classes), then `createNode(..., FLOAT)`.
- Shortly after: 200ms queued `renderTree("window-create")` which does `processFloats()`.
- `processFloats` (tiling-render.ts) calls `isFloatingExempt` (and workspace-skip) for every node and sets `node.float` (which mutates `mode` to FLOAT or TILE).
- `isFloatingExempt` (window.ts:3242) computes `floatByType` including:
  - DIALOG/MODAL types, transient_for, `!allows_resize()`
  - `get_wm_class() === null`
  - `get_title() === null || "" || length===0`
- `get_tab_list` + `findNode(metaMonWs)` paths and reload also involved.
- Recent change (7acda2f): 200ms delay + track only on `workspace::window-added`. `window-created` remained immediate.
- No `notify::wm-class` / `notify::title` listeners; classification only happens on renders.
- Mocks and unit tests asserted the old "null identity ⇒ float" behavior.

## Root Cause

`window-created` (and even delayed workspace-added) can surface a `Meta.Window` before its Wayland app_id / X11 WM_CLASS / title properties have arrived. When the first classification render runs (~200 ms), the null/empty checks in `floatByType` fire, `mode` becomes FLOAT, and the window never participates in tiled layout. Subsequent size/pos-changed renders only re-eval if the window is the focus target at that instant; many apps stabilize identity without an immediate re-render that would "correct" it. Explicit per-app "tile" rules were also defeated at the matcher stage when class was still null.

## Fix

1. `window-created` handler now wraps `trackWindow` in an 80 ms `GLib.timeout_add` (parity with the workspace-added delayed path, gives metadata a head start).
2. Removed the two unstable "identity null/empty" conditions from `floatByType`. Strong float signals (dialog types, transient, `!allows_resize()`, float overrides) remain. Default for a NORMAL window is now "tile" unless a positive reason to float exists. Because `processFloats` + `renderTree` run live on every render, once the real class/title appear the correct decision (including user tile/float rules) is applied automatically.
3. Updated/added unit tests exercising late-null metadata for both `isFloatingExempt` and the "Inkscape-like" path; adjusted expectations that codified the old eager-float rule.
