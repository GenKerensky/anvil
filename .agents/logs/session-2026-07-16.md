# Session 2026-07-16

## Rounded shadow composition and workspace ownership

### Symptom

Using Mutter's native/client shadow with Anvil's rounded surface mask restored square shadow plates
at window corners. Replacing that shadow with a rounded `St.Bin` fixed the corners, but the
free-standing actor stayed visible while switching to another workspace.

### Root cause

The surface effect must discard source shadow pixels because those pixels are composed before the
rounded mask. The replacement shadow was then added directly to `global.window_group`; unlike a
`Meta.WindowActor`, an arbitrary sibling there has no Mutter workspace identity or inherited paint
visibility.

### Fix

`BorderController` keeps the source-shadow crop and owns one replacement shadow per decoration
record. The shadow is now a child of the window's compositor actor, stacked below its surface and
positioned in buffer-local coordinates. It therefore inherits the window's workspace visibility
and actor lifetime while retaining focused/unfocused CSS styles and state reconciliation.

### Validation

- The focused border unit suite passed 27/27, including a regression that rejects a
  `global.window_group` shadow.
- The full `npm test` gate passed: 55 tiling tests, 1,056 unit tests, and 45 tooling tests with the
  two host-only tests skipped as expected.
- The GNOME Shell 50.1 headless border suite passed 5/5 in both legacy and core modes. Its lifecycle
  case switches away from the window workspace and verifies that both the window and child shadow
  become non-paintable, then restore together on return.
- A GNOME Shell 50.3 Devkit stage capture showed the rounded terminal frame and replacement shadow
  at all four corners without a square shadow plate; the isolated session logged no Anvil or GJS
  errors.

## Late-mapped Xwayland decorations

### Symptom

Obsidian and other Electron applications running through Xwayland tiled normally but did not get a
corner mask or active border outline when focused.

### Root cause

Xwayland can emit the workspace admission edge before `Meta.Window.get_compositor_private()`
returns its actor. `WindowTracker` added the valid window to the tree and marked it tracked, but its
later map edge treated tracked legacy windows as complete and only retried admission unmaximize.
The new `MetaWindowActorX11` was therefore never registered with `BorderController`.

### Fix

The legacy tracked-window map path now completes actor admission idempotently: it registers the
actor with `BorderController`, owns destroy and first-frame handlers in `actorSignals`, and
re-registers on first-frame so the mask can attach after the X11 surface exists.

### Validation

- A focused unit regression reproduces a valid window admitted with no compositor actor, then maps
  the actor later and requires its mask, child shadow, and active border.
- A forced-X11 Codex Desktop Electron window reproduced the failure as
  `MetaWindowActorX11` with no decorations before the fix. Afterward, GNOME Shell 50.1 reported a
  masked `MetaSurfaceActorWayland`, parented focused shadow, and visible active border; the border
  suite passed 6/6 during the diagnostic run.
