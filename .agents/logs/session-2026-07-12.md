# Window corner mask alignment

## Symptom

The active-window border used the expected rounded CSS radius, but the masked square client
surface met that border as a short diagonal segment instead of a concentric inner curve. Radius
arithmetic alone did not fix the mismatch.

## Root cause

`WindowCornerMaskEffect` interpreted normalized texture coordinates using the window actor's size
and origin. `ClutterOffscreenEffect` actually renders into a stabilized framebuffer that is three
logical pixels larger on each axis. For an integer-aligned paint volume, Mutter places two of
those pixels on the top/left and one on the bottom/right. The mask circle was therefore shifted two
pixels above and left of the border circle even though its radius was correct.

## Fix

Keep frame bounds actor-local in `BorderController`. During `vfunc_paint_target`, derive the
logical offscreen size from `get_target_size()` and the actor resource scale, shift the bounds by
the leading half of the actual framebuffer padding, and derive the shader pixel step from that
same offscreen size. The correction is isolated behind the mask effect rather than exposed as
border policy.

## Verification

- Added unit coverage for the asymmetric padding at resource scales 1 and 2.
- Rebuilt and captured the square-corner Electron fixture in a nested GNOME devkit session; the
  client crop and active border now share the same corner center.
- `npm test`: 37 files and 936 tests passed.
- `npm run build`: passed before the final visual run.

## Pointer resize snap-back

### Symptom

Dragging a tiled window edge updated the neighboring split while the pointer grab was active, but
releasing the mouse caused the windows to return to equal sizes.

### Root cause

The Grab-Resize session correctly translated the compositor geometry into sibling percentages.
The grab-end Tiling Render then called `processFloats()`, which assigned `node.float = false` to
every participating tiled window. The `Node.float` setter cleared all sibling percentages on every
false assignment, even if the node was already tiled. This erased the resize state immediately
before geometry apply.

### Fix

Make non-floating classification idempotent. Sibling percentages are now reset only on a real
`FLOAT` to `TILE` transition, preserving the existing behavior for newly re-admitted floating
windows without destroying percentages during ordinary renders.

### Verification

- Added a regression covering mouse grab begin, `size-changed` handling, sibling percent updates,
  grab end, and the real final Tiling Render. It failed with unset percentages before the fix and
  passes afterward.
- `npm test`: 37 files and 937 tests passed.
- Resize-focused E2E was attempted in `fedora-devbox`; GNOME Shell 50 and the suite started, but
  Nautilus produced no windows, so the fixture stalled before executing resize assertions.
