# 2026-07-13 — Rounded window shadows and direct devkit launch

## Symptoms

- Masked tiled windows had rounded visible content, but their client-side shadow still exposed
  square blur contours against a white background.
- Launching the Mutter Development Kit from Codex's shell failed with
  `Invalid MIT-MAGIC-COOKIE-1 key` and `Failed to open display`, even though the same launcher
  worked when typed into Ptyxis.

## Root causes

- Clipping the client shadow to a rounded outer envelope removed its square extent but could not
  reshape the square shadow's internal blur contours. The rounded border actor needs to own the
  replacement shadow while the client shadow is clipped away with the window content mask.
- Codex Desktop exports `GDK_BACKEND=x11`. The devkit launcher inherited that override after
  switching its isolated D-Bus activation environment to the nested Xwayland display, where the
  frontend did not own the corresponding Xauthority cookie. A normal terminal did not export the
  override, so GTK selected the outer Wayland display there.

## Fixes

- Apply the corner mask to the complete window actor, discard pixels outside the rounded visible
  frame, and render a rounded box shadow from Anvil's existing border actor.
- Unset `GDK_BACKEND` in `run-devkit-session.sh` before starting the isolated session, allowing GTK
  to select the outer Wayland display while Mutter continues to provide nested Wayland and X11
  displays for test applications.

## Validation

- Focused corner-mask and border-controller unit suites passed (16 tests).
- `make build debug` completed successfully.
- `bash -n` and `git diff --check` passed for the devkit launcher.
- The launcher ran directly from Codex without a terminal wrapper, activated Anvil, opened GNOME
  Settings, and exposed a targetable `Mutter Development Kit` window to Computer Use. The session
  was then stopped cleanly and its diagnostics retained under `/tmp/anvil-devkit.5j9oZK`.

## Distrobox E2E application launch failures

### Symptom

- The headless shell and Anvil reached ACTIVE, but every window-based E2E spec timed out waiting
  for Nautilus. Lifecycle-only specs passed.
- Direct GTK launches reported `Authorization required` and `Failed to open display` even though
  the nested Wayland socket was ready.

### Root cause

- `fedora-devbox` inherited `GDK_BACKEND=x11` from Codex Desktop. D-Bus-activated GTK applications
  therefore selected nested Xwayland, but the runner did not own that display's Xauthority cookie.
- The shared isolated-session helper also published its XDG paths too late, allowing an activated
  settings writer to bind to the outer user profile before `test-mode` was set.

### Fix and validation

- Force `GDK_BACKEND=wayland` in both E2E session environments and publish it to the private D-Bus
  activation environment alongside the discovered display.
- Publish isolated XDG and schema paths before the first settings operation in
  `HeadlessShellSession`.
- The focused tiling suite passed 8/8, the complete E2E suite passed 126/126, and
  `make test-debug-loop-lib` passed 36/36 in `fedora-devbox`.

## Portable tiling shadow parity: ordering and client minimum sizes

### Symptoms

- Three-window swap parity disagreed even though both engines produced valid layouts: individually
  observed admissions appended in the portable core while the legacy Tree inserted after the
  selected window.
- Three-way vertical Nautilus tiling requested 349/349/350px shares, while Mutter applied 380px
  frames because the client minimum was larger.

### Root causes

- Bootstrap sorted window identities before attachment, coupling structural order to lexical
  diagnostic order; incremental facts also lacked a batch insertion cursor.
- `Meta.Window.get_min_size()` reports client dimensions. Nautilus reported 430px, while the frame
  minimum was 380px after removing the 50px client/frame decoration delta.

### Fix and validation

- Preserve snapshot/fact order for structural admission, insert after selection, and keep canonical
  inspection sorting independent. The GNOME shadow adapter now submits multi-window observations as
  one fact batch.
- Add `WindowFact.minimumSize` and normalize it with Mutter's client/frame rectangle conversion
  before applying it to desired portable frames.
- Portable tests passed 49/49, focused adapter tests passed 33/33, and the Fedora Devbox tiling E2E
  suite passed 9/9 with settled parity asserted for two-window, swap, and layout-toggle scenarios.

## Headless pointer-grab soak crash

### Symptom

- A real-grab soak using a Clutter virtual pointer and `Meta.Window.begin_grab_op()` caused the
  headless GNOME Shell process to exit with `SIGSEGV` before Jasmine could write results.
- Passing evdev button code `0x110` first produced an explicit invalid-virtual-button warning;
  switching to `Clutter.BUTTON_PRIMARY` removed that warning but did not prevent the crash.

### Root cause and disposition

- The same fixture crashed under both the portable core writer and the legacy writer, isolating the
  problem to Mutter's headless synthetic-grab automation seam rather than Anvil's grab adapter.
- The unsafe suite was removed from the normal E2E matrix. Repeated real pointer drag/resize remains
  a manual Mutter Development Kit soak item; adapter unit tests and keyboard-resize E2E are not
  claimed as equivalent evidence.

### Related validation

- Core mode passed the complete 128-case real-shell suite twice consecutively before this soak
  experiment.
- New rapid minimize/restore, dynamic workspace create/remove, and late Xwayland map scenarios pass
  in the Fedora Devbox.

## Multi-monitor and lock soak boundaries

### Multi-monitor result

- The E2E harness can now launch one to four persistent virtual monitors. A two-monitor core test
  moves a live window to the second output, switches to mirror mode, restores linear mode, and
  verifies stable portable surface identities plus window/surface referential integrity.
- The portable GNOME adapter samples the monitor manager's current logical topology and ignores a
  stale window monitor index rather than querying an invalid work area during reconfiguration.
- Mutter emits two stale work-area assertions while mirror mode collapses a window-bearing second
  virtual output. The same fixture produces the same assertions with the legacy writer, isolating
  them to the headless compositor transition. Physical output removal remains a manual cutover
  gate.

### Lock/session result

- `Main.screenShield` is `null` in the headless automation Shell, so the harness cannot exercise
  lock-screen actors or GDM authentication.
- Pushing the real Shell session-mode stack into `unlock-dialog` and popping it back to `user`
  succeeds. The extension stays active, removes/restores its indicator, retains portable window
  identity, and reports no core failure.
