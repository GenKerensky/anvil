# Session Log — 2026-06-26

## Goal

Investigate and fix regression: newly opened windows no longer automatically tile when the extension starts. Windows only become tiled if the user disables then re-enables the extension.

Symptom was reproducible in devkit sessions: fresh launch → new apps (terminals, etc.) appear floating at natural size/position. Toggle fixes subsequent launches.

## Root Cause

The live creation path (`"window-created"` → `trackWindow`) had two races that the full-reload path (`enable` → `reloadTree` + `trackCurrentWindows`) did not.

### 1. Invalid monitor / workspace at "window-created" time

Mutter frequently delivers `"window-created"` before the `Meta.Window` has a stable monitor or workspace:

- `metaWindow.get_monitor()` → `-1`
- `metaWindow.get_workspace()` → `null`

`trackWindow` did:

```ts
const winMonitor = metaWindow.get_monitor?.() ?? current;
const winWsIndex = metaWindow.get_workspace?.()?.index?.() ?? active;
const metaMonWs = `mo${winMonitor}ws${winWsIndex}`;
const metaMonWsNode = this.tree.findNode(metaMonWs);
if (!metaMonWsNode) {
  this.reloadTree("no-meta-monws");
  return; // <--- window dropped
}
```

Result: `mo-1ws0` (or similar) had no node → early return. No `Node<WINDOW>` was ever created.

The `"window-added"` (debounced 200 ms on workspace) and `"window-entered-monitor"` handlers only called `updateMetaWorkspaceMonitor`, which only moved _existing_ nodes:

```ts
if (existNodeWindow) {
    if (...) metaMonWsNode.appendChild(...);
}
this.renderTree(from);   // no creation
```

A window that missed the initial `trackWindow` was never added to the tree until a full reload.

### 2. Title / class emptiness forcing FLOAT

`isFloatingExempt` contained:

```ts
const floatByType =
    ... ||
    metaWindow.get_wm_class() === null ||
    windowTitle === null || windowTitle === "" || ...
```

New NORMAL windows often have an empty or null title (and sometimes class) at the moment of the 200 ms `"window-create-queue"` render that calls `processFloats()`. This caused the just-created node (started as `FLOAT`) to be set back to float and stay that way.

A disable+enable cycle calls `trackCurrentWindows` on already-mapped windows that have real titles/classes, so they passed the checks and got tiled.

### Contributing factor

The wl-clipboard ephemeral helper change (b054fed) added an unconditional `frame.width <= 2 && frame.height <= 2` test. Real windows legitimately report tiny placeholder rects at creation time on Wayland.

## Fixes

1. `isEphemeralHelperWindow` (utils.ts)

   - Size-based ephemeral detection only for completely anonymous windows (no `wm_class` **and** no title).
   - Known helpers by name still rejected.

2. `isFloatingExempt.floatByType` (window.ts)

   - `wmClass === null` still forces float.
   - Empty title only forces float when combined with null class.
   - Normal windows with a class (even if title arrives late) are no longer auto-floated at first render.

3. `trackWindow` mon/ws lookup (window.ts)

   - Explicit sanitization:

     ```ts
     let winMonitor = metaWindow.get_monitor?.();
     if (typeof winMonitor !== "number" || winMonitor < 0) winMonitor = current;
     let winWsIndex = ...;
     if (typeof winWsIndex !== "number" || winWsIndex < 0) winWsIndex = active;
     ```

   - Prefers the window's own values when valid.

4. Backup creation path (window.ts)

   - In `updateMetaWorkspaceMonitor`:

     ```ts
     if (!existNodeWindow) {
       this.trackWindow(global.display, metaWindow);
       return;
     }
     ```

   - By the time "added"/"entered-monitor" fire, the window has a real monitor+workspace, so the correct mon node exists and creation succeeds.

These changes make the live path robust against the creation-time property races while preserving the "ignore true ephemeral helpers" intent.

## Architectural Notes

- `Meta.Window` properties (monitor, workspace, title, class, frame rect) are **not stable** at the `"window-created"` signal.
- The mon/ws node lookup + creation logic must either sanitize aggressively or have a reliable secondary creation path on the "settled" signals.
- Size-only heuristics for "this is not a real window" are fragile; identity (class/title) should be required when possible.
- `reloadTree` + `trackCurrentWindows` is the "eventual consistency" path; live signals must not drop work that the reload path would have done.

## Follow-up Ideas (not implemented)

- Consider connecting a one-shot `"notify::title"` or using the first real `size-changed` to re-evaluate float status for windows that were created with missing metadata.
- Make the mon node lookup always succeed by ensuring every workspace node has a "current" monitor child even for not-yet-assigned monitors (more complex).
