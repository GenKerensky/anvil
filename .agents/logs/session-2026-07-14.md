# Native shadow ownership and decoration lifecycle

## Symptom

Anvil's per-window `cornerShadow` actors could outlive minimized or destroyed windows and could be
stacked above unrelated windows. Focus changes also refreshed the complete decoration tree, causing
visible desktop darkening.

## Root cause

Anvil duplicated Mutter/client shadow ownership with separately stacked Clutter actors. The partial
focus optimization still left multiple lifecycle authorities and a floating-focus path that upgraded
work to a full border refresh. During the rewrite, destroyed compositor actors also remained briefly
reachable through decoration records, and surface-target destroy signals initially lacked an explicit
disconnect path.

## Fix

- Removed custom shadow actors, CSS, preferences, parsing utilities, compatibility fields, and tests.
- Kept singleton focus/split hints and targeted focus-only scheduling.
- Attached the rounded mask to Mutter's window-surface child because `Meta.ShapedTexture` is
  `Clutter.Content`, not an effect-owning actor; pixels outside frame bounds now pass through.
- Added explicit mask-target replacement, destroy-signal ownership, unmanaged/actor-destroy cleanup,
  and fail-open target validation.
- Kept floating-window focus raises on the active-only decoration refresh path.

## Verification

- `npm run typecheck:e2e`
- `npm test` (55 tiling tests, 1,027 unit tests)
- GNOME 50 headless E2E: borders 5/5, focus 4/4, minimize 3/3
- GNOME 50 Devkit with overlapping Nautilus/Ptyxis lifecycle: focused-window capture showed rounded
  native corners without a square shadow plate; minimize/close completed without Anvil disposed-actor
  or mask warnings. Raw captures are in `.agents/logs/local/` and remain untracked.

# Monitor-churn native crash isolation

## Symptom

The isolated two-monitor churn spec passed, but running it after the full E2E suite crashed the
nested GNOME Shell with status `-11`. Mutter first reported an invalid logical-monitor lookup.

## Root cause

Mutter 50.1's headless `ALL_MIRROR` transition is unsafe after accumulated window move/resize
activity. A GDB run showed the crash in `meta_monitor_manager_get_logical_monitor_neighbor()`, called
from `meta_display_get_monitor_neighbor_index()` through a GJS signal emitted by
`meta_window_move_resize_internal()`. The crash remained when Anvil was disabled at the switch
boundary, proving that the retained failing state belongs to Mutter rather than Anvil. Resident
windows and deferred window actors widened the failure, but draining them did not make an
accumulated Shell process safe.

## Fix

- Run monitor churn only when explicitly tagged and execute legacy/core variants in fresh nested
  Shell processes from a dedicated E2E target.
- Keep secondary-monitor and mirrored-topology window coverage, but close resident windows and drain
  their actors before collapsing or restoring the topology.
- Disable Anvil across Mutter's unsafe transition window, then re-enable it on each settled topology
  and verify that runtime surface state is reconstructed correctly.

## Verification

- Isolated legacy and core monitor churn completed without a native crash; Mutter still emitted its
  known headless mirror-transition monitor/background criticals.
- Core monitor churn passed its portable surface integrity checks after each topology settled.
- The full `npm test` gate passed.
- The dedicated fresh-process E2E target is the final gate for this change.
